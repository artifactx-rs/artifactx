# Operations: backup, restore, rollback

ArtifactX is **stateless and file-based** — there is no database, no daemon
state, no hidden store. A repository *is* its directory. That makes operations
boring on purpose: back it up with `tar`, restore by extracting, recover from a
bad publish with one command.

## The one thing to understand: metadata is derived

The repository root holds two kinds of data:

| Kind | Path | Source of truth? |
| --- | --- | --- |
| **Inputs** | `arx.toml`, `keys/`, `apt/pool/`, `yum/<repo>/<arch>/*.rpm` | **Yes** — author-provided |
| **Derived** | `apt/dists/`, `yum/<repo>/<arch>/repodata/` (incl. `.states/`) | No — regenerated by `arx publish` |

Everything a client downloads (`Packages`, `Release`/`InRelease`, `repomd.xml`,
the `*.xml.gz` streams, signatures) is **deterministically regenerated** from the
pool + signing key + `arx.toml`. So if the *derived* tree is ever lost or
corrupted, you do not restore it — you re-run `arx publish` and it comes back
identical (a fresh state; the live symlink flips atomically, ADR-0004/0008).

This is the whole operational story: **protect the inputs; the rest rebuilds.**

## Backup

### Minimal (recommended) — back up the inputs

```bash
tar czf arx-backup.tgz \
  arx.toml keys/ apt/pool/ \
  $(find yum -name '*.rpm' -printf '%P\n' 2>/dev/null | sed 's#^#yum/#')
# simpler if the yum tree is small: just include all of yum/ except repodata
tar czf arx-backup.tgz arx.toml keys/ apt/pool/ \
  --exclude='yum/*/*/repodata' yum/
```

Smallest possible footprint; restore + `arx publish` rebuilds all metadata.

### Full — back up the whole root (inputs + derived + rollback states)

```bash
# -p preserves the dist/repodata symlinks that point into .states/.
tar czpf arx-full.tgz arx.toml keys/ apt/ yum/
```

Use the full backup when you want to preserve the existing rollback history
(`.states/`) and avoid even a brief republish. **Preserve symlinks** (`tar -p`;
GNU/BSD `tar` keep symlinks as symlinks by default — do not dereference them).

### The signing key is the crown jewel

`keys/private.asc` cannot be regenerated. If you lose it you can still serve the
current repo, but you can never sign an update that existing clients (who trust
the matching public key) will accept. Back it up **separately and encrypted**:

- Use a passphrase-encrypted key at rest (`ARX_KEY_PASSPHRASE` /
  `--passphrase-file`; see the signing docs) so the backup isn't plaintext.
- Store the key backup somewhere different from the pool backup.
- `keys/` is git-ignored on purpose — never commit it.

## Restore

```bash
mkdir restored && cd restored
tar xzpf /path/to/arx-backup.tgz      # -p keeps perms + symlinks

# If you took a MINIMAL backup (no metadata), regenerate it:
arx publish                            # rebuilds apt + yum metadata, signs it

arx serve                              # back online
```

A full backup is already serveable after extraction; the `arx publish` step is
only needed for a minimal (inputs-only) backup.

## Rollback vs restore — pick the right tool

| Situation | Tool |
| --- | --- |
| A publish went out wrong; the previous index was fine | `arx rollback` (atomic pointer flip to the prior state) |
| A package is bad and must be withdrawn | `arx rm <name> [--version]` then `arx publish` |
| The repo directory/disk is lost | restore from backup (above) |
| Metadata is corrupt but the pool is intact | `arx publish` (regenerates everything) |

```bash
arx history                 # list retained states per target
arx rollback                # flip back to the previous state
arx rollback --to <id>      # flip to a specific retained state
```

`gc` never prunes a pool file that a retained rollback state still references, so
rollback stays valid after garbage collection (ADR-0008). Retained states are
bounded (newest few per target); they are a short-horizon safety net, **not** a
backup — for long-term recovery, take real backups as above.

## Why this design (charter)

One binary, no database, deterministic output, atomic publish, derived metadata:
backup is `tar`, restore is `tar x`, recovery is `arx publish`, and a bad release
is one `arx rollback` away. Operations should be boring.
